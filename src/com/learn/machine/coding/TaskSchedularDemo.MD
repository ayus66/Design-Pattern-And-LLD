# Task Orchestrator (DAG Execution Engine) ‚Äî Low Level Design

## Problem Statement

Design a DAG-based task execution engine that validates the graph for cycles, runs independent tasks in parallel, resolves dependencies dynamically, enforces per-task timeouts, retries failed tasks with exponential backoff, propagates failure/skip signals downstream, and supports resuming a partial run from a snapshot.

## Core Algorithm

### DAG Validation ‚Äî Kahn's Algorithm

`Dag.validate()` uses the same in-degree-peeling topological sort that the scheduler uses at runtime. If `processed != tasks.size()`, tasks still with `inDegree > 0` are in (or downstream of) a cycle.

```
1. Build in-degree map for all tasks.
2. Enqueue all zero-in-degree tasks.
3. Dequeue ‚Üí decrement children's in-degree ‚Üí enqueue any newly-zero children.
4. If enqueued count < total tasks ‚Üí cycle detected; report stuck tasks.
```

---

### Execution Flow

```
run(dag, snapshot)
  ‚Üí validate() ‚Äî throws DagValidationException on cycle
  ‚Üí mark snapshot-COMPLETED tasks; count remaining toResolve
  ‚Üí compute unresolved-dependency count per pending task
  ‚Üí submitTask() for every PENDING task with remaining=0
       ‚Üí executor thread: executeWithRetries(td)
            ‚Üí loop until success or retries exhausted:
                 runWithTimeout(td, attempt)
                   ‚Üí submit action to executor
                   ‚Üí future.get(timeoutMillis) ‚Üí success / TimeoutException / ExecutionException
                 on timeout ‚Üí TIMED_OUT (terminal, no retry)
                 on exception ‚Üí backoff (100ms √ó 2^(attempt-1)) ‚Üí retry
                 on retries exhausted ‚Üí FAILED
            ‚Üí returns terminal TaskState
       ‚Üí onTaskResolved() ‚Äî counts down latch, cascades to children
  ‚Üí latch.await() ‚Äî blocks caller until all tasks resolved
  ‚Üí buildResult()
```

## Failure Propagation

When a task terminates with a non-COMPLETED state, `onTaskResolved` computes what signal to send downstream using the task's `continueOnFailure` flag:

| Task state | `continueOnFailure` | Signal to children |
|---|---|---|
| FAILED / TIMED_OUT | `false` | **FAILED** ‚Äî child becomes FAILED without running |
| FAILED / TIMED_OUT | `true` | **SKIPPED** ‚Äî child is skipped, not marked as failed |
| COMPLETED | ‚Äî | no negative signal; child runs normally when all parents complete |

`worseOf(a, b)`: `FAILED` outranks `SKIPPED` outranks `null`. A child with two parents ‚Äî one sending FAILED and one sending SKIPPED ‚Äî becomes FAILED.

---

### Resume from Snapshot

```java
ExecutionResult result = orchestrator.execute(dag);
// fix the transient issue, then:
ExecutionResult resumed = orchestrator.resumeFrom(dag, result.toSnapshot());
```

Tasks already `COMPLETED` in the snapshot are treated as done; their children's unresolved-dependency counts are pre-decremented accordingly. Only the remaining non-completed tasks are submitted.

## Core Classes

### `TaskOrchestrator`

The engine. Uses a single `CachedThreadPool` for both task execution and `runWithTimeout`'s inner `Future` submission.

| Method | Purpose |
|---|---|
| `execute(dag)` | Full run from scratch |
| `resumeFrom(dag, snapshot)` | Partial re-run; skips already-COMPLETED tasks |
| `shutdown()` | Shuts down the executor; call after all runs complete |

---

### `executeWithRetries`

Exponential backoff: `100ms √ó 2^(attempt-1)` ‚Üí 100, 200, 400, 800 ms‚Ä¶

Timeouts are terminal (no retry) ‚Äî a deliberate simplification worth flagging in an interview; a production version might retry timeouts up to `maxRetries` too.

---

### `runWithTimeout`

Submits the action to the executor and calls `future.get(timeoutMillis)`. On `TimeoutException`, cancels the future (sends interrupt to the action's thread) and rethrows. On `ExecutionException`, unwraps and rethrows the original checked exception.

---

### `onTaskResolved`

The cascade engine. Called once per task reaching a terminal state (including tasks that are cascade-resolved without running). Holds `schedulingLock` only for the bookkeeping (remaining-count decrement + next-state decision) ‚Äî never during actual execution or observer notification.

## Concurrency Notes

- **`CountDownLatch`** ‚Äî one count per non-snapshot task; `latch.await()` blocks the caller until the last task resolves.
- **`schedulingLock`** ‚Äî only the dependency-count decrement and next-state branching are serialised; the actual task execution is fully parallel.
- **Defensive executor submission** ‚Äî uncaught `RuntimeException` inside `submit(Runnable)` is swallowed silently by the `Future`; explicit try/catch logs the error and still counts down the latch so the run doesn't hang forever.
- **`ConcurrentHashMap` for state/remaining/worst** ‚Äî multiple tasks finishing simultaneously update independent keys without contention.
- **Interrupt propagation** ‚Äî `runWithTimeout` cancels the future with `mayInterruptIfRunning=true`; `executeWithRetries` catches `InterruptedException` on backoff sleep and returns `FAILED` cleanly.

## Extension Points

- **Task priorities** ‚Äî add a `priority` field to `TaskDefinition`; use a `PriorityBlockingQueue` in the executor to run higher-priority tasks first.
- **Timeout per attempt vs total** ‚Äî add a `totalTimeoutMillis` alongside `timeoutMillis`; track wall-clock start in `executeWithRetries`.
- **Progress callbacks** ‚Äî add a `TaskListener` (Observer) for `onStart`/`onRetry`/`onComplete`/`onFail` events without modifying `TaskOrchestrator`.
- **Distributed execution** ‚Äî replace `ExecutorService` with a remote task queue (SQS, Celery); workers pull tasks and report back via the same `onTaskResolved` callback.
- **Conditional branching** ‚Äî add an `OutputContext` to `TaskAction` and a `Predicate<OutputContext>` on edges; `onTaskResolved` evaluates the predicate before marking a child as ready.
